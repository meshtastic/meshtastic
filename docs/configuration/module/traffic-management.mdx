## Overview

Previous traffic reduction efforts in Meshtastic have mostly been client-side: configure nodes to send less frequently. While effective when every node cooperates, a single misconfigured node broadcasting its position every 30 seconds or hundreds of undecryptable packets from nodes on a different channel can still burden an entire mesh. Traffic Management moves enforcement to the router level so infrastructure operators can protect shared airtime.

The Traffic Management module provides server-side packet inspection and traffic shaping for Meshtastic routers and nodes. It reduces channel utilization by filtering duplicate position broadcasts, rate-limiting chatty nodes, dropping undecryptable packets, responding to NodeInfo requests from a cache, and controlling hop propagation for telemetry and position data.

:::info
Traffic Management must be enabled at both compile time (via `HAS_TRAFFIC_MANAGEMENT=1` in the device's `variant.h`) and at runtime (via the `enabled` config flag). Other devices require custom firmware builds.
:::

The module uses a per-node cache to track position fingerprints, packet counts, and timestamps. Each entry is 10 bytes. On ESP32 devices with PSRAM, the cache is allocated in PSRAM to preserve heap for other uses.

| Platform | Cache Entries | Memory |
| :------: | :-----------: | :----: |
| ESP32 (with PSRAM), Native | 2048 | ~20 KB (PSRAM) |
| nRF52 | 1024 | ~10 KB (heap) |


## Traffic Management Module Config Values

### Enabled

Master enable for the Traffic Management module. When disabled, all traffic management features are inactive and packets pass through unmodified. Default: `false`.

### Position Dedup Enabled

When enabled, the module tracks position broadcasts from each node and drops duplicate position packets that fall within the same grid cell and time window. This prevents nodes that repeatedly broadcast the same location from consuming airtime. Default: `false`.

### Position Precision Bits

Number of bits of precision (0-32) used to truncate latitude and longitude for position deduplication comparison. Lower values create larger grid cells, meaning positions farther apart are considered "the same."

Default: `24` (~10m grid cells).

| Bits | Approximate Grid Size | Use Case |
| :--: | :-------------------: | :------: |
| 32 | ~1 cm | No truncation |
| 24 | ~10 m | Default (pedestrian/general) |
| 20 | ~40 m | Vehicle |
| 16 | ~600 m | City block |
| 12 | ~10 km | Regional |

### Position Min Interval Secs

Minimum time in seconds between position broadcasts from the same node at the same truncated location before a new position is accepted. If a node sends a position that maps to the same grid cell within this window, the duplicate is dropped.

Default: `86400` (1 day). Set to `0` to disable the time-based check.

### NodeInfo Direct Response

When enabled, the node responds directly to NodeInfo requests using cached user data from the local NodeDB. The response is sent as a single-hop reply (`hop_limit=0`) instead of allowing the request to propagate further through the mesh. This is particularly useful on well-placed routers that already have a large NodeDB from hearing traffic across the mesh. A node that might be 2-3 hops from the requestor can answer immediately from cache rather than letting the request ripple outward.

Default: `false`.

### NodeInfo Direct Response Max Hops

Maximum hop distance from the requestor at which this node will respond to NodeInfo requests. The effective value is clamped based on the device role:

| Device Role | Maximum Allowed | Behavior |
| :---------: | :-------------: | :------: |
| `ROUTER`, `ROUTER_LATE`, `CLIENT_BASE` | 3 | Responds to nodes within configured hops (up to 3) |
| All other roles (clients) | 0 | Responds to direct neighbors only |

When set to `0`, the role-based maximum is used (3 for routers, 0 for clients). For routers, setting this to `2` limits responses to nodes within 2 hops. Values above the role limit are clamped down.

Default: `0` (use role-based limit).

### Rate Limit Enabled

Enable per-node rate limiting. When enabled, the module tracks packet counts per node within a sliding time window and drops packets from nodes that exceed the configured threshold. Routing and admin packets are exempt from rate limiting

Default: `false`.

### Rate Limit Window Secs

Time window in seconds for rate limiting calculations. Each node's packet count resets after this window expires. This window is also used by unknown packet filtering if `drop_unknown_enabled` is active (falls back to 60 seconds if this value is `0`).

Default: `0`.

### Rate Limit Max Packets

Maximum number of packets allowed from a single node within the rate limit window. Packets beyond this count are dropped.

Default: `0`.

### Drop Unknown Enabled

When enabled, tracks packets that cannot be decrypted (from nodes on different channels or using different encryption keys) and drops them after the threshold is reached. The counter resets after the rate limit window or 60 seconds if no rate limit window is configured.

Default: `false`.

### Unknown Packet Threshold

Number of undecryptable packets from a single node before subsequent packets from that node are dropped.

Default: `0`.

### Exhaust Hop Telemetry

When enabled, forces `hop_limit=0` on relayed telemetry broadcasts from other nodes, preventing them from propagating beyond the immediate neighbors of this node. The node's own telemetry packets are unaffected.

Default: `false`.

### Exhaust Hop Position

When enabled, forces `hop_limit=0` on relayed position broadcasts from other nodes, preventing them from propagating beyond the immediate neighbors of this node. The node's own position packets are unaffected.

Default: `false`.

### Router Preserve Hops

When enabled, preserves the hop limit for router-to-router traffic instead of decrementing it. Only applies when the local device has a router role (`ROUTER`, `ROUTER_LATE`, or `CLIENT_BASE`). Position and telemetry packets are always excluded from hop preservation (they have their own `exhaust_hop` controls). This extends effective mesh range through infrastructure routers for messages, NodeInfo, and other non-position/telemetry traffic, similar to Zero-Cost Hops but without requiring the favorite node mechanism.

Default: `false`.

#### CLI

:::info

All Traffic Management Module config options are available in the Python CLI.

:::

|                    Setting                     |   Acceptable Values   |        Default         |
| :--------------------------------------------: | :-------------------: | :--------------------: |
|        traffic_management.enabled               |   `true`, `false`     |        `false`         |
| traffic_management.position_dedup_enabled       |   `true`, `false`     |        `false`         |
| traffic_management.position_precision_bits      | `integer` (0-32)      |      `24` (~10m)       |
| traffic_management.position_min_interval_secs   |      `integer`        |    `86400` (1 day)     |
| traffic_management.nodeinfo_direct_response     |   `true`, `false`     |        `false`         |
| traffic_management.nodeinfo_direct_response_max_hops | `integer` (0-3) |  `0` (role default)    |
|     traffic_management.rate_limit_enabled       |   `true`, `false`     |        `false`         |
|    traffic_management.rate_limit_window_secs    |      `integer`        |          `0`           |
|    traffic_management.rate_limit_max_packets    |      `integer`        |          `0`           |
|     traffic_management.drop_unknown_enabled     |   `true`, `false`     |        `false`         |
|   traffic_management.unknown_packet_threshold   |      `integer`        |          `0`           |
|    traffic_management.exhaust_hop_telemetry     |   `true`, `false`     |        `false`         |
|     traffic_management.exhaust_hop_position     |   `true`, `false`     |        `false`         |
|    traffic_management.router_preserve_hops      |   `true`, `false`     |        `false`         |

:::tip

Because the device will reboot after each command is sent via CLI, it is recommended when setting multiple values in a config section that commands be chained together as one.

```shell title="Example:"
meshtastic --set traffic_management.enabled true --set traffic_management.position_dedup_enabled true --set traffic_management.position_precision_bits 24
```

:::

##### Examples of CLI Usage

```shell title="Enable the module"
meshtastic --set traffic_management.enabled true
```

```shell title="Disable the module"
meshtastic --set traffic_management.enabled false
```

```shell title="Enable position deduplication with 16-bit precision (~600m grid)"
meshtastic --set traffic_management.position_dedup_enabled true --set traffic_management.position_precision_bits 16
```

```shell title="Set minimum position interval to 1 hour"
meshtastic --set traffic_management.position_min_interval_secs 3600
```

```shell title="Enable NodeInfo direct response on a router (responds up to 2 hops away)"
meshtastic --set traffic_management.nodeinfo_direct_response true --set traffic_management.nodeinfo_direct_response_max_hops 2
```

```shell title="Enable rate limiting: max 10 packets per node per 60 seconds"
meshtastic --set traffic_management.rate_limit_enabled true --set traffic_management.rate_limit_window_secs 60 --set traffic_management.rate_limit_max_packets 10
```

```shell title="Drop undecryptable packets after 5 from the same node"
meshtastic --set traffic_management.drop_unknown_enabled true --set traffic_management.unknown_packet_threshold 5
```

```shell title="Exhaust hops on relayed telemetry and position broadcasts"
meshtastic --set traffic_management.exhaust_hop_telemetry true --set traffic_management.exhaust_hop_position true
```

```shell title="Enable router hop preservation"
meshtastic --set traffic_management.router_preserve_hops true
```

```shell title="Get the Traffic Management Module Configuration"
meshtastic --get traffic_management
```
