The serial module config options are: Enabled, Echo, Mode, Receive GPIO, Transmit GPIO, Baud Rate, Timeout, and Override Console Serial Port. Serial Module config uses an admin message sending a `ConfigModule.Serial` protobuf.

This is an interface to talk to and control your Meshtastic device over a serial port. The module can be set to different [modes](#mode), each fulfilling a different use-case.

## Serial Module Config Values

### Enabled

Enables the serial module.

### Echo

If set, any packets you send will be echoed back to your device.

### Mode

Defaults to 'Simple'.

Available Values:

- `DEFAULT` equals `SIMPLE`.
- `SIMPLE` operates as a dumb UART tunnel. What goes in will come out, requires a channel named 'serial'.
- `PROTO` exposes the Protobuf Client API on this serial port. You can use this to connect from another device, see the [Arduino client library](https://github.com/meshtastic/Meshtastic-arduino) and the [API Reference](/docs/development/device/client-api).
- `TEXTMSG` will allow you to send a string over the serial port to the device, which will be broadcasted as a text message to the default channel. Any text message received from the mesh will be sent to the serial port as follows: `<Short Name>: <text>`.
- `NMEA` will output a NMEA 0183 Data stream containing the internal GPS or fixed position and other node locations as Waypoints (WPL).
- `CALTOPO` will output NMEA 0183 Waypoints (WPL) every 10 seconds for all valid node locations, to be consumed by [CalTopo / SARTopo](/docs/software/integrations/caltopo.mdx).
- `WS85` will parse and compute wind speed data received over serial from the Ecowitt WS85. Transmits values via telemetry packet every 5 minutes. Modification of the WS85 weather sensor is required (https://hackaday.io/project/196990-meshtastic-ultrasonic-anemometer-wx-station). If option is not available in app then enable with cli command "meshtastic --set serial.mode 6"
- `VE_DIRECT` VE.Direct is a serial protocol used by Victron Energy products. See [this guide](https://beta.ivc.no/wiki/index.php/Victron_VE_Direct_DIY_Cable)
- `MS_CONFIG` is used to configure and view some parameters of [MeshSolar](https://heltec.org/project/meshsolar/)
- `LOG` will output a short, formatted log of packets sent or received by the node, including text, and telemetry packets.
- `LOGTEXT` will output a short, formatted log of every text packet sent or received by the node.

:::tip
`PROTO` mode will show no obvious output when passively observed by a serial monitor. When testing, consider using the [Python CLI](https://github.com/meshtastic/python) with the `--listen` option, to view the stream of protobufs.
:::

### Receive GPIO Pin

Set the GPIO pin to the RXD pin you have set up.

:::tip
With the RAK4631 on the RAK19007 and some versions of the RAK19003 baseboards, the default pins are TXD1 (16) and RXD1 (15). For this device, setting [GPS Mode](/docs/configuration/radio/position/#gps-mode) to `Not_Present` is necessary. For other versions of the RAK19003 baseboard, the pins TXD0 (20) and RXD0 (19) are accessible.
:::

### Transmit GPIO Pin

Set the GPIO pin to the TXD pin you have set up.

:::tip
Connect the TX pin to the other device's RX pin, and vice versa. Connect their grounds to each other (not necessary if they're both plugged into the same USB power source.)
:::

### Baud Rate

The serial baud rate.

### Timeout

The amount of time to wait before we consider your packet as "done".

### Override Console Serial Port

If set to true, this will allow Serial Module to control (set baud rate) and use the primary USB serial bus for output. This is only useful for NMEA and CalTopo modes and may behave strangely or not work at all in other modes. Setting TX/RX pins in the Serial Module config will cause this setting to be ignored.

:::tip
Once module settings are changed, a **reset** is required for them to take effect.
:::

## SenseCAP T1000-E Serial Setup

The [SenseCAP Card Tracker T1000-E](/docs/hardware/devices/seeed-studio/sensecap/card-tracker) is a credit-card-sized Meshtastic tracker running a Nordic nRF52840. It fully supports the serial module via two connection methods described below.

### Pogo Pin Connection (TEXTMSG, SIMPLE, LOG, LOGTEXT modes)

The T1000-E exposes its debug UART (Serial2) on the four pogo pins at the bottom edge of the device. Connect a **USB-to-UART adapter** (FTDI FT232, CP2102, CH340, or similar — 3.3 V logic level) using the following wiring:

| T1000-E Pogo Pin | USB-UART Adapter |
| :--------------: | :--------------: |
| GND              | GND              |
| TX (P0.16)       | RX               |
| RX (P0.17)       | TX               |
| 3V3              | *(do not connect — device is self-powered)* |

:::tip
Always cross-connect TX → RX and RX → TX between the two devices. Use a 3.3 V logic-level adapter; do **not** use a 5 V adapter, as it will damage the nRF52840.
:::

Once wired, configure the module using the T1000-E's USB CDC port (the magnetic charging cable) and set the GPIO pins to the pogo-pin UART:

```shell title="T1000-E TEXTMSG mode — one-shot config"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode TEXTMSG \
  --set serial.baud BAUD_DEFAULT
```

After the device reboots, open the **FTDI adapter port** at `38400 8N1` to send and receive messages.

### USB-Direct Connection (NMEA and CalTopo modes only)

For NMEA and CalTopo output modes, the T1000-E can stream data over its built-in USB CDC port without any extra hardware. Set `override_console_serial_port` to redirect output to the console USB port:

```shell title="T1000-E NMEA mode via USB"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.override_console_serial_port true \
  --set serial.mode NMEA
```

Then open the **same USB port** at `38400 8N1` to receive NMEA 0183 sentences.

:::caution
`override_console_serial_port` is **only** valid with `NMEA` and `CALTOPO` modes. Using it with `SIMPLE`, `TEXTMSG`, `PROTO`, `LOG`, or `LOGTEXT` will produce a configuration error and the serial module will not start.
:::

## Mode Examples

### TEXTMSG — Send and Receive Text Messages

`TEXTMSG` mode bridges the serial port to the Meshtastic text message channel. Any string sent to the serial port is broadcast as a text message; any received text message is printed as `<ShortName>: <message>`.

**Step 1 — Configure the T1000-E** (replace `<USB_PORT>` with your device port):

```shell title="Configure TEXTMSG mode"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode TEXTMSG \
  --set serial.baud BAUD_DEFAULT
```

**Step 2 — Open the pogo-pin UART** (`<FTDI_PORT>`) with a terminal at `38400 8N1`:

### SIMPLE — Transparent UART Tunnel

`SIMPLE` mode (also the default `DEFAULT`) is a raw binary pass-through. Bytes written to the serial port are sent as a mesh packet on a channel named **serial**; received packets on that channel are written as raw bytes back to the serial port.

:::note
A channel named `serial` must exist on your device. Create one with:
```shell
meshtastic --port <USB_PORT> --ch-add serial
```
:::

```shell title="Configure SIMPLE mode"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode SIMPLE \
  --set serial.baud BAUD_DEFAULT \
  --set serial.timeout 250
```

Connect at `38400 8N1` and send up to **237 bytes** per packet. The module treats the end of a packet as a `serial.timeout` millisecond gap in the byte stream (default 250 ms).

---

### PROTO — Protobuf Client API

`PROTO` mode exposes the full [Meshtastic Protobuf Client API](/docs/development/device/client-api) on the serial port. This is the same protocol used by the phone apps and Python CLI. It produces no human-readable output — use the [Python CLI](https://github.com/meshtastic/python) with `--listen` to observe the stream:

```shell title="Configure PROTO mode"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode PROTO
```

```shell title="Listen to protobuf stream on FTDI port"
meshtastic --port <FTDI_PORT> --listen
```

For embedded integration, use the [Meshtastic Arduino client library](https://github.com/meshtastic/Meshtastic-arduino).

### NMEA — GPS Position Stream (USB or Pogo Pins)

`NMEA` mode outputs NMEA 0183 sentences on the serial port every 2 seconds: a `$GPGGA` sentence for the device's own GPS position, plus `$GPWPL` (Waypoint Location) sentences for each node in the mesh that has a valid position.

**Option A — USB direct (recommended for T1000-E)**:

```shell title="Configure NMEA mode via USB"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.override_console_serial_port true \
  --set serial.mode NMEA \
  --set serial.baud BAUD_DEFAULT
```

After the device reboots, open `<USB_PORT>` at `38400 8N1` to read the stream.

**Option B — Via pogo pins**:

```shell title="Configure NMEA mode via pogo pins"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode NMEA \
  --set serial.baud BAUD_DEFAULT
```

**Reading NMEA output:**

### CALTOPO — Search and Rescue Mapping

`CALTOPO` mode sends NMEA `$GPWPL` Waypoint Location sentences every 10 seconds for **all** nodes in the mesh that have a valid position. This is designed for [CalTopo / SARTopo](/docs/software/integrations/caltopo.mdx) to display mesh nodes on a rescue map in real time.

```shell title="Configure CalTopo mode via USB"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.override_console_serial_port true \
  --set serial.mode CALTOPO \
  --set serial.baud BAUD_DEFAULT
```

Open `<USB_PORT>` in CalTopo's GPS input at `38400 8N1`. Each node with a position will appear as a waypoint, refreshed every 10 seconds.

---

### LOG — Mesh Packet Activity Logger

`LOG` mode outputs a formatted log line for every packet the node sends or receives — text messages, telemetry, position, and more. This is ideal for **logging mesh activity to a serial data logger** such as a [SparkFun OpenLog](https://www.sparkfun.com/sparkfun-openlog.html) connected to the T1000-E pogo pins.

```shell title="Configure LOG mode"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode LOG \
  --set serial.baud BAUD_DEFAULT
```

Open the **FTDI port** at `38400 8N1` to watch the stream. Example output:

```
[TX] TEXT  ch=0  from=!deadbeef  to=!ffffffff  "Hello mesh!"
[RX] POSN  ch=0  from=!cafebabe  to=!ffffffff  lat=47.6062 lon=-122.3321
[RX] TELE  ch=0  from=!cafebabe  bat=87%  temp=22.3C
```

---

### LOGTEXT — Text Message Logger

`LOGTEXT` mode logs only text messages (channel messages and direct messages) sent or received by the node. Useful when you only care about human-readable chat traffic:

```shell title="Configure LOGTEXT mode"
meshtastic --port <USB_PORT> \
  --set serial.enabled true \
  --set serial.txd 16 \
  --set serial.rxd 17 \
  --set serial.mode LOGTEXT \
  --set serial.baud BAUD_DEFAULT
```

Output format mirrors TEXTMSG receive output: `<ShortName>: <message>`.

## Python Scripting Examples

The [Meshtastic Python package](https://github.com/meshtastic/python) can configure and interact with the serial module programmatically. Install it first:

```shell
pip install meshtastic
```

### Configure Serial Module via Python

```python title="configure_serial.py"
import meshtastic
import meshtastic.serial_interface

# Connect to T1000-E via USB CDC port
iface = meshtastic.serial_interface.SerialInterface(devPath="<USB_PORT>")

# Configure serial module for TEXTMSG mode via pogo pins
iface.localNode.moduleConfig.serial.enabled = True
iface.localNode.moduleConfig.serial.txd = 16       # P0.16 pogo-pin TX
iface.localNode.moduleConfig.serial.rxd = 17       # P0.17 pogo-pin RX
iface.localNode.moduleConfig.serial.mode = 3       # TEXTMSG = 3
iface.localNode.moduleConfig.serial.baud = 0       # BAUD_DEFAULT = 38400

iface.localNode.writeConfig("serial")
print("Serial module configured. Device will reboot.")
iface.close()
```

### Send a Text Message via Serial (TEXTMSG mode)

Once the module is configured and the FTDI pogo-pin adapter is connected:

```python title="send_via_serial.py"
import serial
import time

# Open the FTDI pogo-pin adapter port (NOT the USB CDC port)
with serial.Serial("<FTDI_PORT>", baudrate=38400, timeout=1) as ser:
    message = "Hello from serial module!\n"
    ser.write(message.encode("ascii"))
    print(f"Sent: {message.strip()}")
    time.sleep(0.5)
```

### Receive Text Messages via Serial (TEXTMSG mode)

```python title="receive_via_serial.py"
import serial

# Open the FTDI pogo-pin adapter port
with serial.Serial("<FTDI_PORT>", baudrate=38400, timeout=5) as ser:
    print("Listening for mesh messages... (Ctrl-C to stop)")
    while True:
        line = ser.readline().decode("ascii", errors="replace").strip()
        if line:
            print(f"Received: {line}")
```

### Read NMEA Stream and Parse Positions

```python title="read_nmea.py"
import serial

# For USB-direct NMEA mode, open the USB CDC port
# For pogo-pin NMEA mode, open the FTDI port
with serial.Serial("<PORT>", baudrate=38400, timeout=2) as ser:
    print("Reading NMEA sentences... (Ctrl-C to stop)")
    while True:
        line = ser.readline().decode("ascii", errors="replace").strip()
        if line.startswith("$GPGGA") or line.startswith("$GPWPL"):
            print(line)
```

## Interfacing PIR Sensor With External Microcontroller

The following are examples of connecting a Raspberry Pi Pico or Arduino Mini Pro to a PIR motion sensor. When motion is detected, a message is sent via serial to the device. The device transmits the message as text over the default channel using the serial module in `TEXTMSG` mode.

### T1000-E Configuration

- Serial module enabled, mode: `TEXTMSG`
- GPIO Pins: TXD 16, RXD 17 (pogo-pin UART)
- Baud: 38400

### Raspberry Pi Pico BOM

- A Raspberry Pi Pico running [CircuitPython](https://learn.adafruit.com/getting-started-with-raspberry-pi-pico-circuitpython)
- SenseCAP T1000-E (or T-Beam V1.1) running Meshtastic
- PIR Sensor ([Adafruit Breadboard Model](https://www.adafruit.com/product/4871))

### Raspberry Pi Pico Wiring

![image](/img/modules/Serial/pico-pir-wiring.webp)

**For T1000-E (pogo-pin UART):**
- TX pogo pin (P0.16) → RX (GP1) on the Pico
- RX pogo pin (P0.17) → TX (GP0) on the Pico
- GND pogo pin → GND pin 38 on the Pico

**For T-Beam V1.1 (GPIO header):**
- TX pin 14 on the T-Beam → RX pin 2 on the Pico
- RX pin 13 on the T-Beam → TX pin 1 on the Pico
- GND pin on T-Beam → GND pin 38 on the Pico

PIR sensor connections (both boards):
- PIR Vcc → Pico 3V3 (pin 36)
- PIR GND → Pico GND (pin 38)

### CircuitPython Code

```python title="pir_to_mesh.py"
import time
import board
import busio
import digitalio

# Setup PIR sensor on GP28
pir = digitalio.DigitalInOut(board.GP28)
pir.direction = digitalio.Direction.INPUT

# Setup serial UART — TX=GP0, RX=GP1 (cross-connect to device RX/TX)
uart = busio.UART(board.GP0, board.GP1, baudrate=38400, timeout=0)

while True:
    if pir.value:
        uart.write(bytes("Motion Detected\n", "ascii"))
        time.sleep(30)  # debounce — wait 30s before next alert
    time.sleep(0.5)
```

### Arduino Sketch

```cpp title="pir_to_mesh.ino"
// Send a text message over serial when PIR detects motion
// Baud rate must match Meshtastic serial module config (default 38400)

const int PIR_PIN = 2;
unsigned long lastTrigger = 0;
const unsigned long DEBOUNCE_MS = 30000; // 30s between alerts

void setup() {
  Serial.begin(38400);
  pinMode(PIR_PIN, INPUT);
}

void loop() {
  if (digitalRead(PIR_PIN) == HIGH) {
    unsigned long now = millis();
    if (now - lastTrigger > DEBOUNCE_MS) {
      lastTrigger = now;
      Serial.println("Motion Detected");
    }
  }
  delay(500);
}
```
