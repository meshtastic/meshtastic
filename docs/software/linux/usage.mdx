This page explains how to run and manage the `meshtasticd` daemon on Linux, including command-line usage, systemd service management, and Docker deployment.

## Default installation locations

When installed via the Linux packages, `meshtasticd` uses the following default paths:

- **Daemon binary**  
  `/usr/bin/meshtasticd`

- **systemd service file**  
  `/usr/lib/systemd/system/meshtasticd.service`

- **Default virtual filesystem (VFS)**  
  `/var/lib/meshtasticd/.portduino/default`

- **Primary configuration file**  
  `/etc/meshtasticd/config.yaml`

- **Additional configuration overrides**  
  `/etc/meshtasticd/config.d/`

Device-specific configuration templates can be enabled by moving them from:

```
/etc/meshtasticd/available.d/
```

to:

```
/etc/meshtasticd/config.d/
```

To edit your default configuration file, simply open it with your text editor of choice, make desired changes and save.  Then restart the daemon.

```shell
sudo nano /etc/meshtasticd/config.yaml
sudo systemctl restart meshtasicd
```

## meshtasticd CLI options

The daemon accepts several command‑line arguments that control how it starts and where it stores its data:

- `-c CONFIG_FILE` – Path to the configuration file to load.
- `-d DIRECTORY` – Directory used as the virtual filesystem (VFS) where node data and state are stored.
- `-e` – Erase the virtual filesystem before starting the daemon.
- `-h MAC_ADDRESS` – MAC address to assign to this virtual node instance.
- `-p TCP_PORT` – TCP port used to expose the Meshtastic API server.

### Example CLI usage

Run `meshtasticd` (as user meshtasticd) locally with a configuration file and a directory for the virtual filesystem on port 4404:

```shell
sudo -u meshtasticd meshtasticd -c config.yaml -d ./data -p 4404
```

This example:

- Loads configuration from `config.yaml`
- Stores node data in the `./data` directory
- Exposes the Meshtastic TCP API on a non-default port `4404`

The TCP API can then be accessed by clients such as the Python CLI:

```shell
meshtastic --host localhost:4404 --info
```

## Monitoring and troubleshooting

These commands can help when troubleshooting or monitoring a running `meshtasticd` instance.

### View service logs (systemd)

If `meshtasticd` is running as a systemd service, you can view logs with:

```shell
journalctl -u meshtasticd
```

Follow logs in real time:

```shell
journalctl -u meshtasticd -f
```

### Check service status

```shell
systemctl status meshtasticd
```

### Restart the service

```shell
sudo systemctl restart meshtasticd
```

### Stop or start the service
```shell
sudo systemctl stop meshtasticd
sudo systemctl start meshtasticd
```

### Running multiple meshtasticd instances

You can run multiple `meshtasticd` instances using the systemd template service. This is useful when attaching bots to different client APIs or when running multiple radios on a single Linux machine. If UDP networking is enabled (`network.enabled_protocols 1`), the instances can communicate with each other as if they were connected via RF.

1. Create a configuration file using the following naming pattern:

```
/etc/meshtasticd/config.d/config-<instance>.yaml
```

`<instance>` can be any name you choose to identify that instance. For example:

```
sudo cp /etc/meshtasticd/config.d/config.yaml /etc/meshtasticd/config.d/config-mediumfast.yaml
```

Add `APIPort` under the `General` section and specify a non‑default port (the default is `4403`):

```text
General:
  APIPort: 4404
  MaxNodes: 200
  MaxMessageQueue: 100
  ConfigDirectory: /etc/meshtasticd/config.d/
  AvailableDirectory: /etc/meshtasticd/available.d/
#  MACAddress: AA:BB:CC:DD:EE:FF
#  MACAddressSource: eth0
```

You may wish to run meshtasticd in simulation mode if you do not have a physical radio attached:

```text
Lora:
  Module: sim
```

2. Start the corresponding systemd instance:

```shell
sudo systemctl start meshtasticd@mediumfast.service
```

The instance name (`mediumfast` in this example) must match the suffix used in the configuration filename.

To access this instance of `meshtasticd` from client apps or the CLI, use the IP address and port specified in your configuration file.

```text
meshtastic --host 192.168.0.69:4404 --info
```

## Usage with Docker

If you do not own a Linux machine, or you just want to separate things, you might want
to run the application inside a docker container.

### Usage

First, [Pull the image](/docs/software/linux/installation/?os=docker)

To run a container, type

```shell
docker run -p 4403:4403 --restart unless-stopped --name meshtasticd meshtastic/meshtasticd:beta
```

or, to get an interactive shell on the docker created container:

```shell
docker run -it -p 4403:4403 --restart unless-stopped --name meshtasticd meshtastic/meshtasticd:beta bash
```

### CH341 USB

If you want to use a `CH341` USB adapter (MeshStick), you need to pass the `bus` device to the container with the `--device` flag:

First, locate the USB device with `lsusb`:

```shell
lsusb
```

You should see something like this:

```plaintext
Bus 001 Device 001: ID 1d6b:0002 Linux Foundation 2.0 root hub
Bus 001 Device 002: ID 1199:9091 Sierra Wireless, Inc. EM7565-9
Bus 001 Device 003: ID 8087:0032 Intel Corp. AX210 Bluetooth
Bus 001 Device 004: ID 13d3:56b2 IMC Networks Integrated Camera
Bus 001 Device 005: ID 06cb:009a Synaptics, Inc. Metallica MIS Touch Fingerprint Reader
Bus 001 Device 006: ID 1a86:5512 QinHeng Electronics CH341 in EPP/MEM/I2C mode, EPP/I2C adapter
```

Locate the `CH341` device and take note of the **Bus** and **Device** numbers. In this example: `001/006`.

Then, run the container with the `--device` flag,
additionally mounting a local configuration file:

```shell
docker run -it -p 4403:4403 --restart unless-stopped --name meshtasticd --device=/dev/bus/usb/001/006 -v ./config.yaml:/etc/meshtasticd/config.yaml:ro meshtastic/meshtasticd:beta
```

### Executing the application interactively

To use Python CLI against exposed TCP port 4403,
type this in the host machine:

```shell
meshtastic --info --host localhost
```

You can interact with the container in the same way as a normal device using a client that supports TCP connections.

### Stop the container

If started with the `--name meshtasticd` arg, you can stop the container with the following command:

```shell
docker kill meshtasticd
```

Else, run this to get the ID:

```shell
docker ps
```

Then, stop the container with command:

```shell
docker kill <id>
```

> Tip: you can just use the first few characters of the ID in docker commands
